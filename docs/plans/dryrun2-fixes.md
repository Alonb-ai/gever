# תוכנית — תיקוני dry-run #2 (#1–4) + קיבוע tunnel

ארבעה באגים מהתמלול החי של dry-run #2 ("רוטשילד→ביצ'יקלטה"), + נעילת פתרון ה-tunnel.
כל שינוי כירורגי. הפרסונה product-critical — בדמות, קצר, בלי רשימות do/don't.

## #1 🔴 carryover — שיחה ישנה דולפת לסשן חדש
**מה קרה:** dry-run #2 נפתח בלהתייחס להזמנת רוסטיקו הישנה ("מחכה לאישור שלהם").
**שורש:** ה-fresh-page (`_chat_for`) פותח "דף חדש" רק ב-gap>3h (`_last_seen` *בזיכרון*) או
`_reset_next` (*בזיכרון*). **שניהם נמחקים ב-restart** → אחרי restart `_last_seen` ריק →
`stale = last is not None ...` = False → אף פעם לא fresh → התורות הישנות מ-Supabase
מתחזרות. בנוסף, ריקון (recon) שהגיע ל-pending ואז ננטש לא "סוגר" את הסשן.
**מטרה:** בקשת הזמנה חדשה לא יורשת סשן הזמנה קודם שהסתיים/ננטש — גם אחרי restart.
**seam (מינימלי, המבצע יבחר):** (א) להתמיד חותמת זמן ב-`prefs._chat` (`{turns, ts}`) ולהשתמש
בה ל-staleness כש-`_last_seen` ריק (שורד restart). (ב) לסמן את השיחה "סגורה" (flag מותמד
ב-`_chat`) כשהזמנה מגיעה למצב terminal/ננטשת (done/failed/card, *או* כשמתחילה הזמנה חדשה
שונה) → התור הבא נפתח fresh. לבחור את הצירוף המינימלי שגורם למקרה שנצפה להיפתח נקי.

## #2 🔴 סטטוס מזויף
**מה קרה:** המשתמש החליף מסעדה ("תנסה יין בכרם"), גבר ענה "ביצ'יקלטה לא הסתדר לי
במערכת אז אני מריץ על יין בכרם" — המציא כשל שלא קרה.
**מטרה:** גבר לעולם לא ממציא סטטוס/סיבה. כשהמשתמש מחליף — פשוט זורם איתו ("סבבה עוברים
ל-X") בלי להמציא למה הקודם "לא הסתדר".
**seam:** חיזוק חוק הפרסונה הקיים ב-`intent.py` ("לא ממציא, שואל") במשפט קצר: לא להמציא
סיבה/תוצאה למשהו שלא קרה; המשתמש שינה דעתו → פשוט להמשיך איתו.

## #3 🟡 guard מול החלפה — מילים ≠ מציאות
**מה קרה:** המשתמש החליף 3 מסעדות, גבר אמר "מריץ קפה בכרם", אבל ה-double-fire guard
(state=="working") חסם בשקט ובפועל רצה ביצ'יקלטה (הראשונה).
**מטרה:** כשהזמנה בתהליך והמשתמש מבקש אחרת — גבר כן (אני עוד על X, רגע מסיים), לא טוען
שמריץ את החדשה.
**seam:** (א) לשמור את שם המסעדה ב-`_booking[phone]` כשנכנסים ל-state="working" (היום
`info=""`). (ב) להעשיר את ה-`_truth_note` של "working": לנקוב במסעדה שבתהליך ולומר למודל
שאי-אפשר להתחיל הזמנה חדשה עד שזו נגמרת. (סופרסיד/ביטול אמיתי = עתידי.)

## #4 🟡 email לא נאסף ב-onboarding
**מה קרה:** onboarding תפס שם אבל לא מייל (המודל ביקש רק מה שצריך לבוקינג).
**מטרה:** onboarding אוסף מייל מוקדם (חסרונו גרם להמצאה/MISSING:email).
**seam:** ניואנס ב-`ONBOARDING_BLOCK` (intent.py) שמדגיש מייל ספציפית, בדמות, לא טופס
("במיוחד המייל — תצטרך אותו כדי לסגור"). ה-MISSING:email נשאר כרשת ביטחון.

## קיבוע tunnel (נעילת הפתרון הקבוע)
ה-doc קיים (`docs/ops-tunnel.md`: ngrok dev domain). נועלים בקוד כדי שאיש לא יחזור
ל-localhost.run: `scripts/tunnel.sh` שמריץ `ngrok http 8000 --url https://$NGROK_DOMAIN`
(קורא מ-.env), עם שגיאה מנחה אם `NGROK_DOMAIN` ריק. להוסיף `NGROK_DOMAIN=` ל-.env.example,
ולהפנות אליו מ-CLAUDE.md/README במקום ssh localhost.run.
