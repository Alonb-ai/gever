## עיקרון מנחה (החלטת מוצר, 2026-07-02)

**הניווט הוא לא-דטרמיניסטי בכוונה.** המודל מנווט בזכות עצמו (browser-use), בלי
מתכונים קשיחים per-אתר. כשהוא לא יודע מה לעשות או חסר לו נתון — הוא **עוצר ושואל
את המשתמש** (פרוטוקול `MISSING:<field>` הקיים), לא ממציא. חוקי הברזל (לא כרטיס
אשראי, לא לזייף סטטוס, עצירה לפני commit בלי אישור) הם החוזה הקבוע; כל השאר גמיש.
זה מה שמאפשר לגבר להתרחב מעבר למסעדות בלי לכתוב playbook לכל אתר.

## תיעדוף בתוך הזמנת מסעדה (בטא סגורה)

1. **Ontopo / Tabit** (קיים; Ontopo › Tabit) — פלטפורמות הזמנה מוכרות, flow צפוי.
2. **האתר של המסעדה עצמה** — fallback כשאין את המסעדה בפלטפורמות. אותו agent,
   אותם markers; ה-resolve יחזיר `platform="site"` עם דף המסעדה. (Phase 4 של
   `multi-platform-resolve.md`.)
3. **טפסי אונליין כלליים** — האופק שאחרי הבטא. אותה תשתית: פרופיל (שם/מייל/טלפון)
   + `MISSING` כששדה לא מוכר. שום דבר לבנות עכשיו — התשתית כבר הנכונה.

## הערכת סטאק מול החזון

| שכבה | היום | פסק דין |
|---|---|---|
| ערוץ | WhatsApp · Meta Cloud API | ✅ נכון. הקהל שם, אין אפליקציה לבנות. |
| שיחה | gemini-3.5-flash | ✅ נכון (אילוץ עברית מדוברת, הוערך מול חלופות). |
| ניווט | browser-use + gemini-3-flash-preview | ✅ נכון לחזון — לא-דטרמיניסטי by design. |
| דפדפן | Chrome מקומי → **Browserbase** | ✅ להפעיל (B1) — פותר CAPTCHA/stealth *וגם* מפשט deploy. |
| resolve | DDG HTML scraping | 🟡 מספיק לבטא; שביר לפרודקשן → search API (Brave/Serp) כשיש משתמשים. |
| זיכרון | Supabase + Fernet | ✅ נכון. |
| state הזמנה | dicts בזיכרון התהליך | 🟡 מספיק לבטא; restart באמצע הזמנה מאבד אותה. persist כשיש >5 משתמשים. |

**מה חסר לחזון "מעבר למסעדות":** כלום ארכיטקטוני עכשיו. הגבולות כבר נכונים —
resolve (איפה) / build_task (מה) / markers (חוזה) / MISSING (שאיבת נתונים
מהמשתמש) הם per-task-type באופן טבעי. כשמוסיפים משימה שנייה (תור לרופא, טופס
ממשלתי) — מוסיפים שלישיית resolve+task+שדות, לא משכתבים. להתנגד לפיתוי לבנות
"task framework" לפני שיש שתי משימות אמיתיות.

**שדרוג אחד ששווה תכנון (אחרי הבטא):** פרוטוקול shak-resume — היום `MISSING`
מפיל את הריצה וגבר שואל; הריצה הבאה מתחילה מאפס. בעתיד: session שממתין פתוח
(Browserbase keep-alive) בזמן שגבר שואל את המשתמש וממשיך מאותה נקודה. לא לבנות
עד שכאב ההתחלה-מחדש מורגש בפועל.
