# פיצ'ר: המלצות מודעות-זמינות (Availability-Aware Recommendations)

> תועד 24.7.2026 (בקשת אלון). סטטוס: **מתוכנן — לא בפיתוח.**
> תלות: משלים את תיקון #6 בענף `fix/conversation-layer` (אימות-קיום להמלצות) —
> אותו שלב בזרימה, מרחיבים מ"קיים?" ל"קיים ופנוי?".

## הבעיה

גבר ממליץ על מקומות באזור, הלקוח בוחר, ואז ריצת ההזמנה מגלה שאין זמינות.
מעגל מבוזבז: המלצה → בחירה → 2-3 דקות ריצה → "אין מקום" → חוזרים להתחלה.
ההמלצה חסרת ערך אם אי אפשר לממש אותה. צריך להצליב את תוצאות החיפוש/Maps
עם זמינות אמיתית **לפני** שממליצים — ומהר.

## למה זה קשה

בדיקת זמינות מלאה = ריצת דפדפן (1-3 דק' למקום). על רשימת 3 המלצות זה 5-10
דקות — לא ריאלי לשיחה חיה. צריך נתיב זמינות זול שאינו דפדפן.

## הפתרון המוצע: probe זמינות ברמת HTTP, פר-פלטפורמה

התובנה: ה-resolver כבר מזהה לכל מועמד את הפלטפורמה (ontopo/tabit/...).
לפלטפורמות ההזמנה יש endpoints של זמינות שה-frontend שלהן קורא (JSON,
בלי browser) — אפשר לשאול אותם ישירות:

```
זרימת ההמלצות המורחבת:
חיפוש (Brave) → resolve (קיום + URL + פלטפורמה)   ← קיים היום
             → availability_probe(url, date, party) ← חדש: HTTP בלבד
             → שולחים רק מה שקיים ופנוי (או מסומן "לא נבדק")
```

**כלל הסינון (ההגדרה של אלון):** ה-probe נבדק מול **הבקשה המקורית של הלקוח**
(התאריך/שעה/כמות שכבר נתן בשיחה). מקום שעלה בהמלצות ואין לו זמינות לבקשה
המקורית — **נפסל ויוצא מהרשימה**, לא מוצג עם הסתייגות. הוא גם נכנס לרשימת
"נפסלו בשיחה" (תיקון #5) כדי שלא יחזור. סימון "לא בדקתי זמינות" שמור אך ורק
למקום שעל פלטפורמה שאין לה probe — שם אי אפשר לדעת, וכנות עדיפה על השמטה.

- **ontopo** — ה-endpoint של הזמינות שה-SPA קורא (צריך spike קצר למפות את
  הצורה המדויקת; יש לנו כבר סשנים מוקלטים מולו). קריאת HTTP בודדת ~1-2 שנ'.
- **פלטפורמות אחרות** (tabit וכו') — mapping פר-פלטפורמה, מוסיפים לפי שכיחות.
- **פלטפורמה לא ממופה** → לא חוסמים: ההמלצה נשלחת עם סימון כן
  ("לא בדקתי זמינות שם") — עדיף כנות מהשמטה.
- ה-probes רצים **במקביל** (asyncio.gather, timeout ~4 שנ' כולל) — זה HTTP,
  לא QA דפדפני; אזהרת ה"סדרתי בלבד" לא חלה.

## שמירה על מהירות — תקציב זמן

| שלב | תקציב |
|---|---|
| חיפוש + resolve (קיים) | ~5-10 שנ' |
| 3 probes במקביל | ~2-4 שנ' |
| **סה"כ תוספת** | **~4 שנ' מקסימום** (timeout קשיח; מי שלא ענה → "לא נבדק") |

### fallback ל-UX דו-שלבי (אם ה-probes איטיים בפועל)
לא מעכבים את הרשימה: שולחים המלצות מיד, ה-probes רצים ברקע, והודעת המשך
תוך שניות: "מתוך אלה פנויים הערב ב-22:00: X ו-Y". גם ככה עדיף על המצב היום.

## חיבורים לקיים

- **קלט השעה/תאריך:** אם הלקוח כבר נתן שעה/כמות (כמו בשיחות מ-23.7) —
  ה-probe בודק אותן בדיוק. אם לא — בודקים "הערב, שעה עגולה קרובה, 2 סועדים"
  כברירת מחדל ומצייני זאת.
- **cache קצר:** תוצאת probe נשמרת ~15 דק' פר (מקום, תאריך, כמות) —
  שיחות מרובות באותו ערב לא משלמות פעמיים.
- **רשימת הנפסלים** (תיקון #5): מקום שה-probe מצא מלא נכנס לרשימת
  "נפסלו בשיחה" — גבר לא ימליץ עליו שוב באותה שיחה.

## ספייק נדרש לפני פיתוח (חצי יום)

1. למפות את ה-availability endpoint של ontopo (צורת בקשה/תשובה, האם דורש
   token מהדף). אם הוא דורש רינדור דף — הפתרון נופל ל-fallback הדו-שלבי.
2. לבדוק rate-limit / חסימות על קריאות ישירות (אותה רגישות bot כמו בדפדפן?).
3. למדוד את הזמן האמיתי של 3 probes במקביל מהשרת (פרנקפורט).

## הערות

- קריאה ישירה ל-API לא-רשמי = תלות שברירית (שינוי צד-ספק שובר שקט).
  חייב graceful degradation: כשל probe → "לא נבדק", לעולם לא חסימת ההמלצה.
- Google Places API כמקור המלצות+שעות-פתיחה נשקל ונדחה בשלב זה: עלות +
  dependency חדש, ולא פותר זמינות-הזמנה (רק שעות פתיחה). לחזור אליו אם
  איכות ההמלצות מ-Brave לא מספקת.
